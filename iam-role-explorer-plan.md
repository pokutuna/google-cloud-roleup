# GCP IAM ロールエクスプローラ 計画ドキュメント

作成日: 2026-07-10 / ステータス: 設計合意済み・実装前

このドキュメントは「何を・なぜ作るか」を明らかにするためのものであり、実装手順書ではない。実装セッションはこの文書を設計意図の参照点として使うこと。

---

## 1. 目的

Google Cloud の IAM ロールとパーミッションを「探す・見る・比べる」ための、静的にホストできる SPA を作る。

GCP のロール設計・レビュー・トラブルシュートでは、次のような問いに繰り返し答える必要がある。

- このロールでこの操作はできるのか(例: `bigquery.user` でテーブルの中身は読めるのか)
- ある操作(パーミッション)ができるロールはどれか、その中で最小のものはどれか
- あるサービスにはどんなロールがあるのか
- ロール A と B は何が違うのか、A から B に変えると何を失い何を得るのか
- あるロールと似たロール・包含関係にあるロール・補完関係にあるロールは何か

公式 Console のロールページやドキュメントの permissions reference は一覧としては存在するが、**比較・逆引き・関連探索が弱く、複数の問いをまたいだ探索の導線がない**。このギャップを埋めるのが本ツールの目的。

## 2. 対象ユースケース(スコープ)

以下の4群を対象とする(検討時の分類 A〜D)。

**A. 権限の充足性確認**
- あるロールが特定のパーミッションを含むか(即答: 含む/含まない)
- ある操作に必要なパーミッションを含む最小のロールの特定
- パーミッションからの逆引き(それを含むロール一覧)

**B. 変更インパクトの見積り**
- ロール A → B の付け替えで「失う権限」「変わらない権限」「得る権限」を見る
- 危険パーミッション(`setIamPolicy` 系、`iam.serviceAccounts.actAs` 等)が差分に含まれる場合の気づきを与える

**C. ロールの比較・選定**
- 事前定義ロール同士の差分表示
- 要件に近いロールの探索(類似・包含・補完関係)

**D. 一覧・棚卸しの入口**
- サービス単位のロール一覧
- ロール単位のパーミッション一覧(構造化された表示)

### 非スコープ(明示的にやらないこと)

- **実プロジェクトのバインディング(誰が何を持つか)の表示・分析**。Policy Analyzer / Console の領域であり、認証を伴うため static SPA の設計と相反する。将来拡張の余地としてのみ言及する。
- **IAM ポリシーの変更操作**。本ツールは読み取り専用のリファレンス・探索ツール。
- **Conditions / Deny ポリシー / PAM のシミュレーション**。
- **カスタムロールの取得の自動化**(初期リリースでは対象外。ローカル JSON のインポートを将来拡張とする)。

## 3. 概念モデル(前提となる事実)

UI とデータ設計は次の実態に基づく。

- ロールは公式に3分類: **基本ロール**(`roles/owner|editor|viewer`、全サービス横断)、**事前定義ロール**(`roles/{service}.{name}`、Google 管理)、**カスタムロール**(project/org 配下)。
- **ロールの実体は「パーミッションのフラットな集合」**(`includedPermissions`)。階層構造は持たない。
- 「サービス → ロール」「サービス → パーミッション」の対応は所有関係ではなく、**名前のプレフィックスから導出される索引**にすぎない。
  - 例: `roles/bigquery.admin` は BigQuery のロールとして索引されるが、中身には `resourcemanager.projects.get` など他サービスのパーミッションが混在する。
  - パーミッションは `service.resource.verb` の命名(例: `bigquery.jobs.create`)であり、第1セグメントでサービス索引に、第2セグメントでリソースグループに引ける。
- したがってデータモデルは「ロール → パーミッション集合」の辞書1つを核とし、サービスは**両側(ロール名・パーミッション名)から導出する横断フィルタ**として扱う。画面遷移の軸にはしない。

## 4. UI 設計

### 4.1 設計原則

1. **1画面で完結する**。画面遷移で機能を分けない。ロール選択(左)と文脈表示(右)のマスター・ディテール型。
2. **初期状態を空にしない**。起動直後から全ロールのリストが見えており、操作は「削っていく」方向。検索は絞り込みであって表示の前提条件ではない。
3. **クリックが検索構文を書く**。凡例チップ・サービス見出し・パーミッション行など、あらゆるクリック操作が検索ボックスに qualifier を入力する。構文を覚えなくても使え、使ううちに覚える(GitHub filter / Linear 方式)。
4. **状態は検索ボックス+選択チップの1箇所に集約**され、そのまま URL に載る。ブックマーク・共有・ブラウザバックが自然に機能する。
5. **「見る」と「比べる」を操作として分離**する。行クリック=詳細を見る、チェックボックス=比較に積む。選択数による暗黙のモード切替で迷わせない。
6. 概念(サービス/ロール/パーミッション)の説明は**ページ上部の凡例1行**で行い、凡例自体が操作可能(クリックで qualifier 入力)であることでチュートリアルを兼ねる。

### 4.2 画面構成

```
┌──────────────────────────────────────────────┐
│ 凡例: [s: サービス] の下に [r: ロール] があり │  ← 概念説明+クリック可能チップ
│       その中身が [p: パーミッション]          │
├──────────────────────────────────────────────┤
│ 🔍 検索ボックス(インクリメンタル+qualifier) │
│ 選択トレイ: [r: bigquery.user ×] [r: ... ×]  │
├──────────────────┬───────────────────────────┤
│ 左: ロールリスト │ 右: 文脈ペイン            │
│ (常時表示)      │  - 選択0: 使い方ガイド    │
│ サービス見出しで │  - 選択1: 詳細+関連ロール │
│ グルーピング     │  - 選択2+: 比較(diff)     │
│ ロール行に ☑     │  - p: 選択時: 逆引き      │
└──────────────────┴───────────────────────────┘
```

- 左リストは仮想スクロール前提。サービス見出しは折りたたみ可能。基本ロールは最上部に別枠。
- 右ペインには全幅展開ボタンを設ける(3ロール以上の比較など幅が足りない場合)。

### 4.3 状態遷移(ユーザーフロー)

一本道+選択解除で戻る、のみ。

```
起動(全ロール一覧+使い方) → 絞り込み(検索/チップクリック)
  → 1件クリック(詳細+関連ロール) → 2件目を ☑(比較に切替)
  ← 選択チップの × でいつでも前の状態に戻る
```

ユースケースとの対応:
- **A-1 充足性**: `r:` で検索 → クリック → 詳細ペイン内フィルタで `p:...` → ヒット/ミス即答。ミス時は「含まれない — このパーミッションを含むロール n 件 →」で逆引きに接続。
- **A-2 逆引き**: omnibox で `p:` 検索 → パーミッションを選択(アンカーがパーミッションになる)→ 右ペインに含有ロール一覧(パーミッション数昇順=最小権限順)。
- **B インパクト**: 現ロール選択 → 関連ロールリストの「+比較」→ 比較ペイン。比較ペインに「入替モード」トグルを置き、セクション見出しを「失う/変わらず/得る」に読み替える。
- **C 選定**: 1件選択時点で関連ロールリストが常時表示される。
- **D 一覧**: 検索ボックスに `s:bigquery`(またはサービス見出しクリック)で左リストが絞られる。

### 4.4 検索(omnibox)と qualifier

- 構文: `s:{service}` `r:{role}` `p:{permission}`。すべて部分一致。qualifier なしは3種横断。
- 複数指定は AND(例: `s:bigquery p:tables`)。
- 実装はコマンドパレット系ライブラリ(cmdk 等)を想定。入力中はカテゴリ分けされたサジェスト(サービス/ロール/パーミッション、各エンティティ色つき)を出す。
- **文脈による対象の違い**: omnibox はロール/パーミッションの「選択」のための検索。右ペイン内には別途表示フィルタ入力を置き、選択済みロールのパーミッション行を `s:` / `p:` で絞る。同じ構文・別の入力欄として分離する(1つの欄で文脈依存にすると混乱するため)。

### 4.5 色の体系(2軸を役割分担)

- **エンティティ種別色(固定)**: サービス=teal、ロール=purple、パーミッション=amber。凡例チップ・サジェスト・選択トレイのチップに使う。
- **ロール個体色(比較時に割当)**: 比較に積んだロールごとに系列色(A=purple、B=teal、…)。比較ペインのセクション縁取り・見出し・比率バー、詳細ペインのカバレッジ棒に使う。
- 2軸の衝突を避けるため、種別色はチップ/アイコンに、個体色は比較系の可視化に限定する。

### 4.6 詳細ペイン(1件選択時)

左右2分割。

**左: パーミッションツリー**
- `service.resource` をグループキーに折りたたみ表示。グループ行にロールアップ件数。
- ペイン内フィルタ(`s:` `p:`)で行を絞る。
- 各パーミッションに optional の短い説明文(データがあれば表示、なければ名前のみ)。
- フィルタでヒットなしの場合、「含まれない」警告と逆引きへのリンクを表示する(ミスを行き止まりにしない)。

**右: 関連ロールリスト**
- 選択ロール R に対し、関係ごとにアイコン+カバレッジ棒で表示:
  - `⊃` スーパーセット: R を完全に包含(棒: R 全体を覆い+余剰の薄色セグメント)
  - `⊂` サブセット: R に完全に包含される(棒: 一部のみ、余剰なし)
  - `≈` 部分重複: Jaccard 類似度上位(棒: 共通の濃色+余剰の薄色)
  - `⊥` 非重複・補完候補: 同サービスで共通ほぼゼロ(棒: ほぼ薄色のみ)
- **カバレッジ棒は2セグメント**: 濃色=R との共通部分(R の個体色)、薄色=そのロール独自の余剰。数値は `共通/|R| +余剰` 表記(例: `42/42 +81`)。アイコンを読まなくても棒の形で関係が判別できることを意図する。
- 並び順: 同一サービス優先 → 共通パーミッション数降順。
- 各行に「+比較」アクション。クリックで比較ペインへ。

### 4.7 比較ペイン(2件以上選択時)

**列対照ではなく、所属で行を分けるセクション型 diff** を採用する(2カラム対照表は視線の往復が多く、右カラムの幅でも破綻するため)。

- 上部に**比率バー**(A のみ / 共通 / B のみ の件数比、各セグメントはロール個体色/グレー)。セグメントクリックで該当セクションへジャンプ&フィルタ。
- 本体は3セクションの縦積み: 「A のみ」「共通」「B のみ」。
  - 各セクションはロール個体色の左ボーダー+見出し色で所属を示す。
  - **共通セクションはデフォルト折りたたみ**(diff ビューアの流儀。差分が主役)。
  - セクション内は詳細ペインと同じ `service.resource` グルーピング+ロールアップ。
- **意味バッジ**: パーミッションの verb / 既知パターンに基づき「データ閲覧」「IAM 変更」「削除系」「なりすまし(actAs)」等のバッジを自動付与。差分に危険パーミッションが含まれることを見出しレベルで気づけるようにする(ユースケース B の事故防止)。
- **入替モードトグル**: セクション見出しを「失う(A のみ)/ 変わらず(共通)/ 得る(B のみ)」に読み替える表示モード。
- **3ロール以上**: 所属パターン(2^n−1 通り)をそのまま並べると爆発するため、「全ロール共通」「1ロールのみ」「その他の組み合わせ」に丸めて表示し、「その他」は展開時にパターン別(UpSet の区画に相当)に分ける。件数の大きい区画から表示する。

### 4.8 逆引きペイン(パーミッション選択時)

詳細ペインの亜種(左右の役割を入替):
- 左: 選択パーミッションの説明・所属サービス・同リソースグループの近隣パーミッション。
- 右: そのパーミッションを含むロール一覧。分類(基本/事前定義/カスタム)で区分し、**含有パーミッション総数の昇順**(=最小権限に近い順)でソート。

### 4.9 参考にした既存 UI

- **caniuse.com / MDN 互換性テーブル**: メンバーシップ行列、行グルーピング、sticky ヘッダ。
- **UpSet プロット**: 4集合以上の交差サマリ。比率バー→区画フィルタの操作モデル。
- **GitHub compare / diff**: 2ロール比較の diff 意味論と配色の流用。
- **EC の比較カート**(価格.com 等): 選択トレイで探索と比較を接続するパターン。
- **コードカバレッジ HTML レポート**(istanbul): 「n/m」ロールアップつき折りたたみ。
- **GitHub / Linear の filter 構文 + cmdk**: qualifier 検索と「クリックが構文を書く」導線。

## 5. データ設計

### 5.1 ソースと取得

- **事前定義ロール・基本ロール**: `gcloud iam roles list` で全ロール名 → 各ロールを `describe` して `includedPermissions` / `title` / `description` / `stage` を取得。ロールの短い説明文はここから確保できる(公式ソース)。
- **パーミッションのメタデータ(説明文)**: `gcloud iam roles describe` は名前しか返さない。IAM API `permissions.queryTestablePermissions` のレスポンス(Permission リソースの title/description/stage)を第一候補として検証する。埋まりが不十分な場合、公式 permissions reference ページからの抽出を別テーブルとして JOIN する fallback を用意する。
- **サービス対応表**: ロール名プレフィックスとパーミッション第1セグメントからの導出を基本とし、Service Usage API の `services.list`(API 名・displayName)と突合。合致しないプレフィックス(`resourcemanager` 等)のみ手動オーバーライド表を持つ。手動表は小さく保ち、導出でカバーできる範囲を最大化する。

### 5.2 データスキーマ(案)

静的 JSON。ビルド時に生成し、SPA は読み取りのみ。

```
services.json      [{ prefix, displayName, apiName? }]
roles.json         [{ name, title, description, stage, kind: basic|predefined,
                      servicePrefix?, permissionCount }]
permissions.json   [{ name, servicePrefix, resourceGroup, verb,
                      title?, description?, stage? }]
role-permissions   ロール → パーミッション ID 配列(または bitset)
relations.json     前計算: ロールごとの { supersets[], subsets[],
                      similar: [{role, jaccard, shared}], complements[] }
badges.json        危険/意味バッジの判定ルール(verb・名前パターン → バッジ)
```

- パーミッションは全体で ID 化(整数)し、ロール側は ID 配列で持つ。規模感: 事前定義ロール約 2,000 弱 × パーミッション約 10,000。
- クライアントでは ID 配列からビットセット(Uint32Array)を構築し、包含判定・交差・Jaccard を集合演算で行う。関連ロール(top-k)はビルド時に前計算して JSON に焼くが、比較ペインの任意組み合わせはクライアント計算とする。
- **配信サイズは要実測**(§6)。必要ならサービス単位のチャンク分割+遅延ロード。ただし関連ロール・逆引きは全体データを要するため、まず全体 1 ファイル+gzip/brotli で成立するかを確かめる。

### 5.3 更新パイプライン

- 目的: **データファイルの定期再生成だけで運用が回る**こと。アプリのデプロイとデータ更新を分離する。
- 生成スクリプト(gcloud または REST 直叩き)を CI(GitHub Actions 想定)で日次実行し、差分があれば commit → 静的ホスティングに反映。事前定義ロールの変化は緩やかなので日次で十分。
- 生成時にスキーマ検証と件数の異常検知(前回比で大幅減なら fail)を入れ、上流の変化で壊れたデータを配信しない。

## 6. 実装前に検証すべき事項

1. **パーミッション説明文の入手性**: `queryTestablePermissions` の title/description の実際の埋まり具合。→ 結果次第で permissions reference のスクレイプ fallback の要否が決まる。
2. **サービス対応のカバレッジ**: プレフィックス導出+Service Usage 突合で何割対応が取れるか。手動オーバーライド表に残る件数の実測。
3. **データ量**: 全ロール展開 JSON の生サイズ / gzip 後サイズ。初期ロードに耐えるか、チャンク分割が要るか。
4. (小)`roles describe` の全件取得にかかる時間と API レート。並列度・キャッシュ方針の決定材料。

1〜3 は「gcloud で実データを1回全部吐き出す」ことで同時に検証できるため、実装セッションの最初の作業として推奨する。

## 7. 技術方針

- React SPA、静的ホスティング(ビルド成果物+データ JSON のみ。サーバ・認証なし)。
- 状態管理: 選択(ロール/パーミッションのアンカー)と検索クエリを URL(ハッシュまたはクエリ)にシリアライズ。共有可能・戻る対応。
- リスト表示は仮想スクロール(react-virtuoso 等)。owner 等の巨大ロール(数千パーミッション)でも成立させる。
- 検索: cmdk 等のコマンドパレット系。データ全体がメモリに載る前提なので全文検索ライブラリは不要、単純な部分一致+スコアリングで足りる見込み。
- ダーク/ライト両対応。色はセマンティックトークンで管理し、エンティティ種別色・ロール個体色の2軸(§4.5)をトークンとして定義する。

## 8. 将来拡張(初期リリースには含めない)

- **カスタムロールのインポート**: `gcloud iam roles list --project` の JSON をローカルで読み込ませ、事前定義と同じ土俵で比較・関連探索できるようにする。static SPA のままファイルドロップで実現可能。
- **ロールの組み合わせ比較**: 「dataViewer + jobUser」の和集合を仮想ロールとして比較対象にする(⊥ 補完候補の発見と対になる機能)。
- **principal 視点**: あるプリンシパルのロール集合を入力し、実効パーミッションの合成を見る(バインディング分析には踏み込まず、ロール合成のみ)。
- **変更履歴**: データ更新パイプラインの差分を蓄積し、「このロールにいつパーミッションが増えたか」を引けるようにする。

## 9. 成功の判断基準

- ユースケース A の問い(例: `bigquery.user` で `tables.getData` は可能か)に、**起動から3操作以内・ノーヒント**で到達できる。
- 2ロールの差分から「得る/失う」の一覧と危険バッジの有無が、スクロールなしの初期視野で判断できる。
- データ更新が CI の定期実行のみで完結し、手動メンテはサービス対応オーバーライド表の追記に限られる。
